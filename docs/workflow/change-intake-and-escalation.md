# 变更请求受理、冻结边界与升级规则

## 1. 目标

- 统一定义变更请求的分级判定：可直接执行 / 必须 OpenSpec / 必须 BLOCKER 升级
- 在受理阶段提前拦截范围漂移与契约破坏风险
- 与 Harness 门禁联动，确保“无验证证据不合并”

## 2. 判定检查点（受理时必查）

- `Scope Boundary`：是否仅在授权目录、授权模块内改动
- `Contract Boundary`：是否变更 API、数据结构、错误语义、外部约定
- `Change Boundary`：是否引入范围外功能、跨仓库依赖或额外流程复杂度
- `Verification Readiness`：是否具备可执行 Harness 计划与可追溯证据口径

## 3. 分级判定条件

### 3.1 可直接执行（Direct Execute）

同时满足以下条件，才可直接进入开发：

1. 改动完全位于已授权范围，且不突破冻结边界
2. 不新增功能，不扩展业务范围，不改变既有流程目标
3. 不变更契约（API/数据结构/错误语义）
4. 验收标准明确，且已给出可执行 Harness 计划（命令 + smoke + 证据形式）
5. 风险可在当前 Issue 内闭环，不依赖额外架构决策

### 3.2 必须 OpenSpec

命中任一条件，必须先 OpenSpec，再拆解 Issue：

1. 新增功能或扩展冻结范围
2. 变更契约（接口、字段、状态码/错误语义、上下游约定）
3. 跨仓库或跨团队依赖发生调整
4. 涉及关键架构取舍、回滚策略变化或高不确定性方案
5. 现有验收标准无法覆盖预期行为，需要先定义新规则

### 3.3 必须 BLOCKER 升级

命中任一条件，必须立即升级 `BLOCKER`，并暂停继续推进：

1. 出现边界冲突（请求内容与冻结范围矛盾，且无法降级）
2. 契约冲突导致上下游无法继续联调或验证
3. 存在安全、数据破坏或线上事故风险
4. P0/P1 关键路径阻塞超过 4 小时且无替代路径
5. 缺失验证前置或验证结论不可追溯（无 Harness 计划/无证据/证据失效）

## 4. 与 Harness 门禁联动（强制）

### 4.1 Issue 受理门禁

以下任一缺失，Issue 不得进入 `Ready`：

- 验收标准（DoD）
- Harness 计划（必跑命令、关键 smoke、证据形式）
- 分级结论及触发依据（Direct Execute / OpenSpec / BLOCKER）

### 4.2 PR 合并门禁

以下任一情况，PR 必须判定为 `BLOCKER` 且禁止合并：

- 未提供 Harness 验证证据
- 仅声明“已测试”但无法回放命令、日志或关键结论
- 验证存在失败项且无明确阻断结论与处置记录
- 实际改动突破 Issue 冻结边界但未先更新真源并重走判定

## 5. 升级动作与时限

触发 `BLOCKER` 后，执行以下动作：

1. 新建或标记 `BLOCKER` Issue，明确影响范围与阻断原因
2. @PM + 架构 + 相关 FE/BE 负责人
3. 24 小时内输出处理路径：回退、降级、拆分或补充 OpenSpec
4. 关闭条件：边界冲突解除 + Harness 证据补齐 + 风险结论可审计
