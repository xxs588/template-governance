# OpenSpec 治理变更回写与归档基线

## 目标与范围

本基线用于约束控制面仓库中治理类变更的最小产物、`validate/archive` 准入、以及归档后的主 `specs` 维护规则，确保治理规则可追踪、可校验、可持续演进。

## 1. 治理变更最低产物

### 1.1 proposal（变更提案）

每条治理变更必须提供 `proposal.md`，至少包含：

- 变更动机（Why）与要解决的问题
- 变更范围与非目标（Scope / Non-goals）
- 影响面（受影响仓库、流程、能力）
- 验证影响（需要哪些验证活动）
- 迁移与回滚口径（若适用）

### 1.2 design（设计说明）

每条治理变更必须提供 `design.md`，至少包含：

- 关键设计决策及取舍理由
- 产物映射（proposal/specs/tasks 与仓库落点关系）
- 与现有规则的兼容策略
- 失败回退路径或替代方案

### 1.3 specs（能力规范）

每条治理变更必须提供对应 `specs/*/spec.md` 增量，至少包含：

- 变更后生效的规范性要求（MUST/SHALL 级语义）
- 可验证场景（Scenario）与验收判定
- 边界与约束（包含非目标或不覆盖项）
- 风险与回滚触发条件（若有）

### 1.4 tasks（执行任务）

每条治理变更必须提供 `tasks.md`，任务必须可独立验证，且每个任务至少定义：

- 输入文档（来自 proposal/design/specs）
- 产出物路径
- 验收检查方式（命令、检查点或证据类型）
- 回写目标（归档后需更新的主 specs 或规则文档）

## 2. validate 准入规则

执行 `openspec validate <change-id>` 前，必须满足：

- `proposal/design/specs/tasks` 四类产物齐备且相互可追踪
- 规范要求与任务验收之间存在一一映射关系
- 不存在未处理的 BLOCKER 级冲突（规则冲突、边界冲突、回滚缺失）

`validate` 通过判定：

- 变更产物结构完整且引用路径有效
- 规范场景可被任务或验证证据覆盖
- 变更不引入未声明的治理真源冲突

未通过 `validate` 的变更禁止进入 `archive`。

## 3. archive 准入规则

执行 `openspec archive <change-id>` 前，必须满足：

- `validate` 已通过
- 本次变更要求的主 `openspec/specs/` 回写已完成
- 临时执行性说明已留在变更归档目录，不污染主 specs 正文

归档完成后应满足：

- 归档目录保留完整变更链路（proposal/design/specs/tasks）
- 主 `specs` 只保留当前生效规则，不保留阶段性叙述
- 变更关系可追踪（可识别 supersede/replace 关系）

## 4. 归档后主 specs 维护规则

### 4.1 单一真源维护

- 主 `openspec/specs/` 是生效治理规则真源
- 归档目录用于追溯历史，不作为当前规则入口

### 4.2 回写优先级

- 归档前必须先回写主 `specs`，再执行 archive
- 主 `specs` 表达最终规则，历史推导过程放在归档变更中

### 4.3 变更策略

- 禁止绕过 change 直接改写治理性主 `specs`
- 新一轮规则调整必须创建新 change，并声明与历史变更关系
- 若发生回滚，需通过新 change 明确回滚目标与生效边界

### 4.4 文本稳定性

- 主 `specs` 使用可长期有效的规范语言
- 不写周报、阶段状态、临时责任人、临时里程碑等短期信息
