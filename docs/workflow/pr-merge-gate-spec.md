# PR 合并门禁与验证证据规范（R5）

## 目的与适用范围

本规范定义 PR 合并前的最小门禁集合与验证证据格式，确保评审基于可复核证据而非主观阶段判断。适用于控制面仓库所有面向主干分支的 PR。

## 核心原则

- 门禁规则优先于标签：`BLOCKER` / `REQUIRED` / `FOLLOW-UP` 仅作为记录工具，不能替代合并门禁判定。
- 无证据即未验证：声明“已通过”但缺少证据，按未通过处理。
- 先真源后实现：涉及规则/契约变更时，必须先更新真源文档并在 PR 中给出链接。

## PR 必填字段

以下字段在 PR 描述中必须完整填写：

- `Issue`：关联 issue 链接（可多条）。
- `Spec`：对应规范文档链接；若无新增规范，需填写“沿用 + 链接”。
- `Contract/Rule`：受影响契约、边界或规则清单；无变更需写明“无变更”。
- `Harness`：本次验证项与执行结果摘要，至少包含 `Build/Typecheck`、`Unit Test`、`Smoke`（不适用项需写 `N/A + 原因`）。
- `证据摘要`：对日志/截图结论的可复核摘要，并提供定位信息。
- `风险与回滚`：风险说明与回滚路径。

## 验证证据格式

每个验证项都必须包含以下四部分：

1. `Command`：执行命令或操作入口。
2. `Result`：通过/失败/不适用。
3. `Evidence`：日志片段、截图或报告链接。
4. `Conclusion`：一句话说明与门禁关系。

推荐格式（可直接粘贴到 PR）：

```text
### Build/Typecheck
- Command: <命令>
- Result: PASS | FAIL | N/A（原因）
- Evidence: <日志片段/链接/截图路径>
- Conclusion: <是否满足合并门禁>
```

证据要求：

- 日志类：至少包含命令、关键输出、退出码或成功标识。
- 截图类：需包含时间、页面/模块标识、核心结论标注。
- 链接类：必须可访问并能定位到对应运行记录。

## 阻断条件（Merge Blockers）

出现任一项即禁止合并：

- PR 必填字段缺失或明显不实。
- `Spec` / `Contract/Rule` 与代码变更不一致，且未完成真源更新。
- 必需验证项（Build/Typecheck、Unit Test、Smoke）结果为 `FAIL`，或无有效证据。
- 存在未解决的 `BLOCKER` 评论。
- 存在与目标分支冲突且未解决，或 CI 必须检查未通过。

## Request Changes 条件

评审应给出 `Request Changes`（而非仅评论）的情况：

- 命中任一阻断条件。
- 证据无法复核（链接失效、日志缺关键上下文、截图无法证明结论）。
- “不适用（N/A）”无合理原因，或规避了必须验证项。
- 发现契约破坏、范围越界、回滚不可执行等高风险问题。

## 可批准条件

仅在以下条件全部满足时可 `Approve`：

- PR 必填字段完整且可追溯。
- 必需验证项均通过并有可复核证据。
- 未解决的 `BLOCKER` 为 0，`REQUIRED` 已闭环或转为明确 issue。
- 风险与回滚方案明确且可执行。

## 与标签体系的关系

- 标签用于表达评论性质与优先级，便于沟通与沉淀。
- 合并决策必须依据本规范门禁项与证据状态，不以标签数量或阶段归属替代判定。
