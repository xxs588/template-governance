# 文档仓库本身如何成为控制面

> 面向内部分享和个人思考的初稿。
>
> 这里说的“控制面”指的是整个文档仓库，而不是单独的 `docs/control-plane/` 目录。`docs/control-plane/` 只是这个控制面里的宪法层。

## 1. 先给结论

我现在越来越明确的一点是：

AI Native 项目里，文档仓库如果只是放资料，它最多只是知识库；但如果它能把边界、真源、流程、证据、回写统一起来，它就不再只是“文档存放地”，而会变成整个项目的控制面。

控制面的价值，不是让团队多写文档、多走流程，而是把原本藏在人脑、口头沟通、聊天记录、临场经验里的协作秩序，提前外置成 AI 和人都能共同遵守的工程约束。

如果一句话再压缩：

控制面不是在解决“有没有文档”，而是在解决“项目怎么才能不越做越偏”。

## 2. 这篇文章想回答哪四个问题

为了让整篇更容易读，我想把主线收束成四个问题：

1. 为什么 AI 项目比传统项目更容易越做越偏？
2. 什么情况下文档仓库只是知识库，什么情况下它会升级成控制面？
3. 如果控制面是整个文档仓库，那这个仓库内部应该怎么分层？
4. 这套工作流为什么真的能减少跑偏，而不是只是把流程写得更复杂？

后面的内容，都围绕这四个问题展开。

## 3. 为什么 AI 项目更需要控制面

传统开发里，很多东西其实是靠人补脑补出来的：

- 需求有点模糊，开发会追问
- 文档有点过期，老同学会提醒
- 范围有点扩了，评审时会有人拦一下
- 产品、PM、开发不开会时，真实进度往往是黑箱，但人至少知道该去问谁
- PR 证据不完整，Reviewer 有时还能靠经验大概判断

但 AI 不是这样工作的。

AI 的典型问题通常不是“不会写”，而是：

- 会在错误目标上高速推进
- 会把旧样例、旧目录、旧规则误当成当前真源
- 会顺手扩 scope，把相关但未授权的东西一起做掉
- 会在没有证据的情况下，主观认为“应该差不多好了”
- 会在任务结束后不主动回写，导致经验继续留在聊天记录里

所以，AI Native 项目的核心挑战不是“怎么让 AI 更努力”，而是“怎么让 AI 不在错误方向上越做越快”。

这也是为什么我现在会把重点放在“控制面”上，而不是只放在 Prompt、工具或者单次执行效果上。

## 4. 什么叫“文档仓库升级成控制面”

如果一个文档仓库只能回答“这里有哪些资料”，它本质上还是知识库。

如果一个文档仓库开始稳定回答下面这些问题，它才开始升级成控制面：

- 这次到底做什么，不做什么
- 哪些文档是当前真源，冲突时听谁
- 哪些改动可以直接做，哪些必须升级讨论或走 OpenSpec
- 开发执行前应该读什么，忽略什么
- PR 提交时必须证明什么
- 合并或发布后，哪些状态必须回写

也就是说，控制面管理的不是“信息存放”，而是“决策秩序”。

所以我现在更愿意把控制面理解成下面五件事的统一体：

- 边界
- 真源
- 流程
- 证据
- 回写

只要这五件事还是分散在人脑、聊天和临场判断里，项目就还没有真正进入控制面状态。

## 5. 如果只用一个框架来理解控制面

如果要让我用一个最容易记住的框架来讲，我会把“整个文档仓库作为控制面”拆成四层：

1. 宪法层：定义长期有效的边界和治理红线
2. 流程层：定义项目推进的标准动作和准入规则
3. 验证层：定义什么证据才算真的完成
4. 运行层：定义当前阶段做到哪里、卡在哪里、风险在哪里

把这四层叠在一起，再加上 `templates / contracts / openspec` 这些支撑件，整个仓库才构成完整控制面。

换句话说：

- `docs/control-plane/` 不是控制面本身
- 它只是控制面里的宪法层
- 真正的控制面是整个文档仓库的组合

这一点我觉得必须讲清楚，因为如果一开始就把“控制面”误解成 `control-plane/` 子目录，后面很多理解都会偏。

## 6. 对照我们的仓库，这个控制面具体长什么样

从 `blog-Docs` 这个真实仓库和 `template-governance` 这个抽象模板里，其实已经能看出一个相对完整的控制面骨架。

### 6.1 它首先是一个“分层仓库”，不是一个“文档堆”

固定层负责长期有效的规则：

- `docs/control-plane/`：项目宪法层，冻结范围、契约边界、治理边界
- `docs/workflow/`：流程 SOP，定义任务如何 intake、拆解、下发、评审、发版、回写
- `docs/harness/`：验证闭环，规定什么证据才算验证完成
- `docs/standards/`：模板和编写标准，统一人和 AI 的表达方式

运行层负责高频变化的状态：

- `docs/runtime/phase-current.md`
- `docs/runtime/issue-dispatch.md`
- `docs/runtime/risk-register.md`

支撑件负责让这套控制面真正可执行：

- `templates/`：统一 Spec、Issue、PR、Harness checklist 的表达结构
- `contracts/`：冻结跨端契约真源
- `openspec/`：让重大变更先冻结、再进入执行

所以，控制面并不是某个目录，而是这套“分层 + 真源 + 模板 + 变更机制”的整体。

### 6.2 为什么这种分层这么重要

因为如果没有分层，项目很容易陷入两种坏状态。

第一种坏状态是：固定规则和运行状态混在一起。

结果会是：

- 文档越来越脏
- 阶段状态被误读成长期规则
- AI 把短期叙事当成长期约束

第二种坏状态是：只有规则，没有运行态。

结果会是：

- 规则写得很完整
- 但没人知道现在推进到哪
- 文档仓库越来越像档案馆，而不是调度台

所以固定层和运行层分离，不是为了好看，而是为了让“长期规则稳定”和“短期状态灵活”同时成立。

### 6.3 真源层级为什么是控制面的核心

`blog-Docs` 里已经形成了一套比较明确的权威顺序：

- L0：治理主真源
- L1/L2：专项规则和执行规则
- OpenSpec：治理变更准入真源
- runtime：运行态同步记录

这件事在人类团队里看起来像“文档管理习惯”，但在 AI 协作里其实是基础设施。

因为人看到冲突文档时，通常会追问：

- 哪个更新
- 哪个算数
- 这是不是旧版本

AI 往往不会主动这么做。它更容易把最近读到的、字面最完整的、语气最强的内容当成真相。

所以，真源顺序不是锦上添花，而是防止 AI 抓错依据的硬约束。

### 6.4 为什么入口文档必须薄，深层文档要按需展开

外部方法论有一点我现在非常认同：入口文档不能变成百科全书。

`AGENTS.md`、`README.md`、控制面入口页，更适合承担这些职责：

- 告诉大家这套仓库是做什么的
- 告诉大家应该先读什么
- 指向高权重真源
- 给出最小操作路径

真正的细节，应该下沉到深层文档里按需加载。

这样做不是为了优雅，而是为了控制注意力。

AI 最怕的常常不是“信息不够”，而是“信息权重错了”。

- 入口太厚，AI 会把大量当前任务不需要的信息一起读进来
- 入口太薄但又没有导航，AI 会自己漫游仓库，抓“看起来有用”的材料

好的控制面结构，本质上是在替 AI 设计阅读顺序。

## 7. 这套工作流为什么能减少跑偏

如果把这套工作流压成一条主线，我会这么讲：

讨论收敛 -> OpenSpec 冻结 -> tasks 拆解 -> Issue 下发 -> Agent 执行 -> PR 证据 -> PM 验收 -> 回写控制面 -> 下一轮讨论

它真正的价值，不在于“步骤变多了”，而在于每一步都在防一种特定的偏移。

### 7.1 讨论收敛和 OpenSpec 冻结，防目标漂移

很多项目真正失控，不是因为没人干活，而是因为边界根本没冻结。

如果讨论没有被收敛成 Spec、proposal、design、tasks，后面的开发就只能依赖当下语境理解需求。人越多、轮次越多、AI 参与越深，偏差就越大。

OpenSpec 的作用，就是把“讨论过”变成“冻结了”。

从这一步开始，项目不再围绕聊天推进，而是围绕冻结资产推进。

### 7.2 tasks 和自包含 Issue，防执行入口漂移

Issue 自包含是这套工作流里我认为最 AI Native 的设计之一。

它要求执行入口里直接写清楚：

- 目标是什么
- 真源是什么
- 非目标是什么
- 允许改哪里，不允许改哪里
- Harness 怎么验证

这相当于把“应该给 AI 的上下文”前置收敛成一个单点入口。

它至少带来三个收益：

- 降低对聊天和口头补充的依赖
- 降低换人、换 Agent、跨轮接手时的信息损耗
- 降低 AI 自己漫游仓库、抓错真源的概率

### 7.3 PR 证据和 Harness，防“看起来做完了”

AI 很容易在没有真正验证的情况下给出“已经完成”的感觉。

所以控制面不能只定义流程，还要定义证据。

PR 里要求：

- 关联 Issue
- 关联 Spec
- 关联 Contract
- 给出命令、结果、证据、结论
- 对照 Spec 场景覆盖
- 说明风险和回滚

这一步的本质，是把“我觉得做完了”改成“系统能审计地证明做完了”。

这也是为什么文档层一定要和 Harness 层接起来。

单靠文档，只能让 AI 更倾向于做对；加上 Harness，才能在它做错时暴露、收敛、阻断和回滚。

### 7.4 回写，防组织失忆

很多团队的问题不是没有经验，而是经验从来没有进入控制面。

如果 PR 合并后不更新：

- `issue-dispatch`
- `risk-register`
- 运行态同步记录
- 必要时的新规则或模板

那么下一轮讨论就还是从零开始。

所以回写不是收尾动作，而是控制面开始积累复利的地方。

没有回写，流程只是一次性交付。
有回写，流程才会越来越像系统，而不是一次次重复人工协调。

## 8. 这套工作流比传统 PM -> 开发 -> 验收 强在哪

传统流程更像：

PM 提需求 -> 开发自己理解 -> 开发提交结果 -> 大家一起看像不像

这套控制面工作流更像：

PM 发起讨论 -> 冻结边界 -> 生成可执行任务 -> 指定真源 -> 要求证据 -> 回写系统记忆

差异不在于有没有 PM、有没有开发、有没有验收，而在于治理重心变了。

传统流程主要在管理：

- 谁在做
- 做没做完
- 结果像不像

控制面流程主要在管理：

- 为什么现在做这件事
- 这次做到哪里算完成
- 这件事做到哪里了
- 哪些不能顺手一起做
- 这次是依据什么做的
- 谁能复核它真的做完了
- 下次还能不能沿着这条线继续做

对于 AI 来说，后者比前者重要得多。

因为 AI 最擅长的是“在给定目标下高效展开”，最不擅长的是“自己判断组织语境里哪些隐含约束最重要”。

## 9. 经过这次梳理，我现在最想强调的三个判断

### 9.1 从结构角度看

真正决定这个仓库是不是控制面的，不是文档数量，而是：

- 目录有没有表达职责边界
- 真源顺序有没有讲清楚
- 固定层和运行层有没有分开
- 入口和深层文档之间有没有清晰导航

如果这些没有成立，仓库再大也更像资料堆。

### 9.2 从工作流角度看

真正让开发不容易跑偏的，不是“环节很多”，而是每个环节都形成了可追责抓手：

- OpenSpec 冻结边界
- Issue 自包含执行入口
- PR 证据约束验收
- 回写把经验纳入系统

如果这些环节只是形式存在，流程再完整也只是表演。

### 9.3 从 AI 角度看

AI 最大的问题常常不是产出不够，而是会在不完整理解下高速产出一个自洽答案。

所以对 AI 来说，控制面不是文档美化，而是把原本人类能靠经验兜住的协作习惯，翻译成可读取、可执行、可验证、可回写的外部秩序。

## 10. 我认为当前体系已经做对了什么

结合 `blog-Docs` 和 `template-governance`，我认为当前体系已经做对了五件关键事。

### 10.1 已经不再把文档仓库当资料库

`blog-Docs` 已经明确把自己定义成控制面仓库，而不是普通文档库。这意味着项目开始围绕规则、契约、闭环和回写组织，而不是围绕聊天记录组织。

### 10.2 已经把固定规则和运行状态拆开

这是控制面能长期维护下去的前提。

如果没有这一层拆分，规则会被阶段叙事污染，运行状态又会被错误当成长期约束，AI 读取时会越来越混乱。

### 10.3 已经把治理规则自己也纳入治理

OpenSpec 不只治理业务功能，也治理控制面本身。

这意味着规则不是拍脑袋写进 README，而是通过 proposal、design、specs、tasks 来迭代。只有当治理规则自己也有变更准入，控制面才不是“一份总说明”，而是真正能演进的系统。

### 10.4 已经把回写纳入正式闭环

很多团队只管交付，不管回写；你们已经把回写写进主流程了。

这意味着控制面不只是用来发任务，也用来吸收经验。

### 10.5 已经开始朝模板化复制走

`template-governance` 说明这套方法已经不是一次性的项目文档整理，而是开始沉淀为可以复制到其他项目的控制面框架。

一套方法只有能被复制，才算真正抽象出来。

## 11. 这套体系现在最值得警惕的失控点

根据仓库现状和讨论结果，我认为当前最值得警惕的是下面五类问题。

### 11.1 控制面自身也会漂移

如果 README 路径、AGENTS 索引、契约真源路径、OpenSpec 最小包定义之间出现不一致，控制面本身就会向 AI 发出冲突信号。

这类问题对人来说只是“文档有点旧”，对 AI 来说却可能变成“我到底该按哪套来”。

### 11.2 runtime 容易成为最弱的一层

很多团队能把规则层写完整，却长期不维护运行态面板。

结果就是：

- 固定层越来越完整
- 实际推进越来越依赖口头同步
- runtime 成了模板占位，而不是运行控制台

一旦如此，控制面就会只剩“宪法”，没有“调度台”。

### 11.3 回写很容易在忙碌时被跳过

这也是最危险的一类退化。

因为回写不是立刻产出功能，所以它往往最先被牺牲；但一旦回写缺失，所有偏差、例外、真实工期、真实验证成本都无法进入下一轮控制面。

### 11.4 证据可能流于形式

如果 PR 证据只是勾选框、口头说明、无法复核的截图，那么流程表面完整，实际还是在凭感觉合并。

证据设计的关键不是“有没有”，而是“能不能支撑复核”。

### 11.5 模板仓库可能被误用成“拷贝即完成”

`template-governance` 只是控制面骨架，不是已经填满项目真相的仓库。

如果新项目只复制结构，不补：

- 范围冻结
- 系统一页设计
- 治理规则
- 契约真源
- 各执行仓库 AGENTS

那它看起来像控制面，实际上只是空壳。

## 12. 如果让我拿这篇去做一次分享，我会怎么讲

我会把整场分享收束成下面这个顺序：

### 12.1 第一步：先把问题讲透

不要一上来讲目录结构，而要先讲：

为什么 AI 项目更容易越来越偏？

因为很多原来靠人补脑解决的问题，到了 AI 这里都会被放大。

### 12.2 第二步：再讲什么叫控制面

文档仓库不是为了写更多文档，而是为了把项目的边界、真源、流程、门禁、回写放到同一个可追溯系统里。

讲到这里，再去讲目录结构，大家才会知道每个目录不是“多一个分类”，而是在解决哪一种治理问题。

### 12.3 第三步：讲整个仓库如何构成控制面

这时再去讲：

- 宪法层
- 流程层
- 验证层
- 运行层
- templates / contracts / openspec

让大家先建立整体图，再去看细节。

### 12.4 第四步：最后讲这套工作流为什么能防偏

可以直接用下面这张映射来讲：

| AI 常见弱点 | 控制面对应机制 |
|---|---|
| 扩 scope | Spec / Non-goals / Freeze boundary |
| 抓错真源 | Authority map / Source of truth |
| 上下文漂移 | Issue 自包含 / AGENTS 入口 / Progressive disclosure |
| 主观完工 | PR 证据 / Harness / Scenario 覆盖 |
| 组织失忆 | 回写 / risk register / runtime sync |

### 12.5 第五步：最后再讲演进方向

不要把这套体系讲成“已经完成”，而要讲成“已经形成控制面雏形，接下来持续提高执行质量”。

最现实的优化方向是：

1. 持续校正文档路径、真源定义、入口导航的一致性
2. 把 runtime 从模板态推进为真实运行面板
3. 提高 PR 证据的可复核性，避免空心化
4. 把回写变成默认动作，而不是额外负担
5. 把高频返工场景继续沉淀成 Rules / Skills / SOP

## 13. 最后一句话

文档仓库之所以能成为控制面，不是因为它把“写文档”这件事做得更重，而是因为它把项目推进里最容易丢失、最容易口头化、最容易让 AI 误解的部分，提前固化成了可以被读取、被执行、被验证、被回写的工程秩序。

真正要守住的，不是文档数量，而是这条秩序：

先冻结，再执行；先证据，再合并；先回写，再进入下一轮。
