# 文档仓库作为控制面的写作大纲与思考点

> 用途：这不是正文，而是一份辅助写作的提纲。
>
> 目标不是替你写完，而是帮你先把“我要讲什么、为什么这样讲、每一段真正要想清楚什么”整理出来，方便你按自己的理解继续深化。

## 1. 先明确这篇文章到底要回答什么

在动笔前，建议先把文章收束到 1 个主问题和 3 到 5 个子问题，否则很容易写成“道理很多，但主线不够强”。

### 主问题

为什么说一个文档仓库，在 AI Native 项目里，不只是文档库，而可以成为整个项目的控制面？

### 子问题

1. AI 开发为什么比传统开发更容易越做越偏？
2. 什么叫“控制面”，它和普通知识库、PRD 仓库、规范仓库有什么本质不同？
3. 为什么“控制面”说的是整个文档仓库，而不是 `docs/control-plane/` 一个目录？
4. 这个仓库内部各层分别在解决什么问题？
5. 这套工作流为什么真的能减少漂移，而不是只是增加流程负担？

如果你后面写的时候发现某一段很难归到这些问题里，通常说明那段要么还没想清楚，要么不属于主线。

## 2. 这篇文章最适合的总结构

如果你是想写成“分享文”或者“思考文”，我建议不要一开始就讲目录结构，而是按下面这个顺序来：

1. 先讲问题
2. 再讲定义
3. 再讲框架
4. 再讲工作流
5. 最后讲价值、风险和演进

这个顺序的好处是，读者会先知道“为什么要听你讲这个”，再进入“你这套东西到底是什么”。

## 3. 推荐写作大纲

下面这份大纲不是唯一答案，但很适合作为你后续扩写的骨架。

### 第一部分：我为什么开始重新理解“文档仓库”

这一部分不要急着解释结构，而是先讲你的认知变化。

你可以回答：

- 我以前怎么理解文档仓库？
- 为什么后来觉得“只是写规范/写 README”不够？
- 是哪些真实问题让我意识到，项目推进真正缺的是控制面，而不是更多零散文档？

这一段的作用是把文章从“抽象方法论”拉回“真实问题驱动”。

### 第二部分：AI 项目到底比传统项目多了什么问题

这一部分建议只讲 3 到 5 个最核心的问题，不要铺太散。

可以考虑的点：

- 人类团队很多时候靠补脑、追问、默契、老同学经验来兜底
- AI 没有这种稳定的责任感和上下文记忆
- AI 会在错误目标上高速推进
- AI 会抓错真源
- AI 会顺手扩 scope
- AI 会把“看起来做完了”误当成“已经真的做完了”
- AI 不会天然替你做回写和知识沉淀

你真正要讲清楚的是：

AI 的问题不是“能力不强”，而是“能力太强，但缺少稳定约束，所以会把组织的不清晰放大”。

### 第三部分：什么叫控制面

这部分是整篇文章的概念锚点，最好写得非常清楚。

建议你重点思考：

- 控制面到底在控制什么？
- 它控制的是代码吗？流程吗？决策吗？上下文吗？还是这些的组合？
- 它和普通知识库最大的差异是什么？
- 它和单纯的规范仓库有什么不同？

一个很好的写法，是把“知识库”和“控制面”并列对比。

比如可以思考：

- 知识库回答“这里有什么资料”
- 控制面回答“现在该以什么为准、下一步该怎么推进、做完怎么证明、结果回写到哪里”

如果这一段写清楚，后面整篇会稳很多。

### 第四部分：为什么控制面说的是整个文档仓库

这是你这篇文章里非常关键的一层，因为它直接纠正一个很容易出现的误解：

控制面不是 `docs/control-plane/` 目录，而是整个文档仓库。

这里建议你把整个仓库拆成几个层来讲。

一个比较好记的讲法是四层：

1. 宪法层
2. 流程层
3. 验证层
4. 运行层

再补充三个支撑件：

- `templates`
- `contracts`
- `openspec`

你可以逐层思考：

- 这一层面向谁？
- 它解决什么问题？
- 如果没有这一层，会发生什么失控？

如果你把这一部分写好，读者就会从“看到一个目录树”升级到“看到一个控制系统”。

### 第五部分：这套工作流为什么能让项目不容易越做越偏

这一部分建议围绕闭环来写，而不是围绕文件来写。

主线可以是：

讨论收敛 -> 冻结决策 -> 拆解任务 -> Issue 下发 -> 执行 -> 证据验收 -> 回写 -> 下一轮

每一步建议你都回答两个问题：

1. 它在防什么偏？
2. 如果没有这一步，会发生什么问题？

比如：

- 没有冻结决策，开发入口就会漂
- 没有自包含 Issue，执行上下文就会漂
- 没有 PR 证据，验收就会漂
- 没有回写，组织记忆就会漂

这样文章会更像“偏差控制链路”，而不是“流程说明书”。

### 第六部分：这套体系比传统项目管理强在哪里

这部分很适合写成对比，但不要写成简单表格堆砌，要讲“治理重心变化”。

你可以想清楚：

传统流程主要在管理什么？

- 谁在做
- 做没做完
- 结果像不像

而控制面流程主要在管理什么？

- 为什么现在做
- 做到哪里算完成
- 哪些不能顺手一起做
- 依据是什么
- 谁能复核
- 做完后怎么进入组织记忆

如果你把“治理重心变化”讲透，读者更容易意识到这不是“多一个文档仓库”，而是“多了一套项目推进方式”。

### 第七部分：你现在最认可的价值、最警惕的风险、最想继续优化的点

这一部分最好带有你自己的判断，而不是只做总结。

建议至少回答三类问题：

- 这套体系现在最有价值的点是什么？
- 最容易失控、最容易流于形式的点是什么？
- 如果下一步继续优化，你会优先改哪几个地方？

这部分很重要，因为它会让文章有“你的思考”，而不是只有“结构分析”。

## 4. 写作时最值得深想的几个关键问题

下面这些问题，不一定都要写进正文，但很值得你先想透。

### 4.1 为什么“控制面”这个词比“规范仓库”更准确

你可以思考：

- “规范仓库”更多让人想到静态规则
- “控制面”更强调动态统筹、状态感知、边界管理、执行闭环

如果你认同这点，文章里最好把这个区别明确写出来。

### 4.2 为什么 AI 时代“上下文治理”会变成核心问题

传统工程里，大家更常讲规范、测试、流程。

AI 时代多出来的一层是：

- 给 AI 看什么
- 不给 AI 看什么
- 哪些是高权重真源
- 哪些是旧文档、旧实现、旧样例

这一点其实非常关键，因为它是 AI Native 和传统工程治理最不同的地方之一。

### 4.3 为什么“回写”不是补充动作，而是控制面的核心动作

这是特别值得你深写的一点。

很多人会把回写当成收尾，但你完全可以把它写成控制面最有复利的一层。

你可以思考：

- 为什么没有回写，流程只是一次性交付？
- 为什么有回写，组织才会真的越来越不容易重复犯错？
- 为什么 AI 项目里回写比传统项目更重要？

### 4.4 为什么“证据”在 AI 场景下比在人类场景下更重要

因为 AI 非常擅长给出一个“自洽但未验证”的结果。

所以你可以重点想：

- 什么叫“看起来完成”和“真的完成”的区别？
- 为什么 PR 证据、Harness、Scenario 覆盖是控制面的一部分，而不是开发末尾的小动作？

### 4.5 为什么“整个仓库”比“一个大总文档”更适合作为控制面

这其实是信息架构问题。

你可以想：

- 为什么不能把所有规则都堆到 `AGENTS.md` 或一个 README 里？
- 为什么必须分层？
- 为什么必须有固定层和运行层？
- 为什么必须有入口和深层文档的关系设计？

如果你把这些想清楚，文章的结构感会明显更强。

## 5. 很适合放进文章里的几种图

如果你后面要继续写成分享文，这几种图会很有帮助。

### 5.1 一张总图

主题可以是：

“整个文档仓库如何构成控制面”

图里可以有：

- 宪法层
- 流程层
- 验证层
- 运行层
- templates / contracts / openspec

### 5.2 一张闭环图

主题可以是：

“控制面如何让项目从讨论走到回写”

链路可以是：

讨论 -> 冻结 -> 拆解 -> 下发 -> 执行 -> 验证 -> 回写 -> 下一轮

### 5.3 一张映射图

主题可以是：

“AI 常见失控点和控制面对应机制”

例如：

- 扩 scope -> Scope freeze / Non-goals
- 抓错真源 -> Authority map / Source of truth
- 上下文漂移 -> Issue 自包含 / 入口导航
- 主观完工 -> PR evidence / Harness
- 组织失忆 -> 回写 / runtime / risk register

## 6. 写作时最容易踩的坑

这部分是提醒你写的时候避坑。

### 坑 1：一上来就讲目录

这样读者会知道“有什么”，但不知道“为什么重要”。

### 坑 2：概念很多，但没有一个主判断

如果整篇没有一句强判断，读者最后会觉得“都对，但抓不住重点”。

### 坑 3：把 control-plane 目录误写成控制面本身

这是这篇最需要避免的语义错误。

### 坑 4：只讲方法，不讲为什么你会相信它

如果缺少你自己的判断和观察，文章会很像整理材料，而不是思考文。

### 坑 5：讲了很多优点，但没有讲风险和失败模式

没有风险意识的文章，读起来会像宣传稿，不像真正有经验沉淀的思考。

## 7. 你可以直接拿来扩写的几个句型

当你后面卡住时，可以直接拿这些开头继续写。

### 讲问题时

- 我越来越意识到，AI 项目真正难的不是“写不出来”，而是“很容易在错误方向上持续写下去”。
- 传统团队里很多默认成立的协作能力，到了 AI 这里都不再天然成立。
- 如果边界、真源和验收口径没有被外置，AI 最终放大的不是能力，而是组织的不清晰。

### 讲定义时

- 我理解的控制面，不是一个放规范的地方，而是一套能持续统筹决策、执行、验证和回写的工程秩序。
- 当一个文档仓库开始决定“现在该按什么做、做完怎么证明、结果回到哪里”时，它才真正开始成为控制面。

### 讲结构时

- 这里最重要的一点是，控制面说的是整个文档仓库，而不是 `docs/control-plane/` 这一个目录。
- 目录树表面上是文档分类，实际上表达的是治理分层和协作边界。

### 讲工作流时

- 这套工作流真正强的地方，不是多了几个步骤，而是每一步都在防一种特定的偏移。
- 从讨论到回写，这不是流程拉长，而是偏差被逐段收束。

### 讲结尾时

- 控制面的本质，不是增加文档，而是减少漂移。
- 真正要守住的不是文档数量，而是“冻结、执行、验证、回写”这条秩序。

## 8. 如果你只想先写出一版最小可用正文

那可以只先写下面 6 段：

1. 我为什么开始重新理解文档仓库
2. AI 项目为什么更容易偏
3. 什么叫控制面
4. 为什么控制面是整个仓库
5. 这套工作流怎么防偏
6. 我现在最认可的价值和最警惕的风险

先把这 6 段写透，再决定要不要继续扩到完整分享稿。

## 9. 最后一句提醒

你现在最需要的，可能不是“再补更多点”，而是不断追问每一段：

- 这一段到底在回答什么问题？
- 这一段和上一篇之间是什么关系？
- 这一段如果删掉，主线会不会更清晰？

当你能一直沿着这三个问题去写，整篇文章的整体感通常就会出来。
