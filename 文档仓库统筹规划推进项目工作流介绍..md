## 我为什么开始重新理解“文档仓库”
    这一部分的主要是想把文章从“抽象方法论”拉回“真实问题驱动”。

我以前可能单纯认为文档仓库就是一个放类似于开会讨论或者其他方式出来的所有有关该项目有关的文档，类似于PRD啊，接口规范啊或者细化某些目标或者一些出现有分歧的地方统一之后的决策等等等等，确实如果只从字面上这样子理解，不就是放这些文档吗？对吧，作为知识库，需要的时候我再去找或者直接搬给ai看让他注意就行了，但是这样真的有用吗？ 

后来我发现在实际开发过程中，举个例子吧，我们要开发一个项目，刚开始通过脑暴项目立项，PRD确定，开一些细化需求的会议之后我们得到了一些文档，这都是我们讨论下来具体如何操作如何实现，要达成什么目标，是我们思考后把各种目标，方式，规范在脑子里面刻画出我们对这个项目的“样子”，可能我们清楚了，然后去实现他，具体在开发过程中遇到不确定的问题再通过人讨论之后得出结论产生另一些文档 给我们自己脑子中对这个项目的“样子”改得更细化  然后分模块分工 给具体负责人， 但是每个人心里对产品的样子不可能是完全一样的，每个人对于这个项目这个阶段实际要做什么，要做成什么样子，什么东西不要做，什么东西延后做，肯定都是不清晰的，也就是说可能你想的跟PM 跟产品想的可能都不一样 ，产品经理可能不太清楚开发的实际代码细节和实现是怎么样的，不能给出很好的规范指引具体部分，开发者可能也不擅长产品阶段的目标的冻结，产品阶段边界的收束，验收标准是什么样子，什么应该做，什么不应该这样做，项目主线又是什么，什么是附加项，开发时就很容易跑偏，朝着自己脑子里面的产品努力 

文档都在那，边界和阶段要做成什么样子不是都能定好吗？ 为啥现在开发用AI还是容易越来越偏？

没有明确的收束好边界，没有很好将阶段目标冻结，没有联系真正的文档证据，没有可以追溯的每个具体细节如何实现的真源，再加上每个人将自己心里所想的那个项目样子 模糊描述 给agent ，让他代替你开工，你凭借自己感觉验收，你说可以先写测试或搭很多其他的CI，可是你不还是按照你心
里所想去写的，然后看代码审核按感觉验收的吗  

AI最擅长就是实现，AI通常会不考虑太多，直接以你给的作为真源然后就去实现这个功能，然后他自认为很清晰很完整就去实现，看似效率很高其实和最初越来越偏

（AI的更清晰描述：“当前 AI 执行过程中的核心痛点在于**‘目标偏移’（Goal Drift）**。由于模型倾向于将 Prompt 的字面表述视为绝对真理，而非对其背后意图进行概率推断，导致其在执行过程中缺乏自我校准机制。这种‘语法正确、语境偏差’的现象，使得 AI 在错误的方向上投入了高强度的执行力，最终交付产物虽然符合指令格式，却完全背离了业务初衷。”）

如果你说， 需要添加更多更细致化的文档进去规范他 让AI到时候遇到了去看就行，又或是说单纯写好README AGENTS.md等等 就能起到约束，那你大概率就是在赌，赌AI上下文能够清晰你在说什么事情，赌AI真的会遵从你的规定你的约束，赌“注意力机制”不会走神

所以让我真正意识到需要一个控制面来真正地控制整个项目开发，让每走一步都是有证据，可以溯源的，将具体需求要是什么样子，代码规范要是什么样，有了一个可以参考标准


## 有 AI 参与的项目到底比传统项目多了什么问题
    这部分主要是列一下比较常见的问题

- 人类团队很多时候靠补脑、追问、默契、老同学经验来兜底
- AI 没有这种稳定的责任感和上下文记忆
- AI 会在错误目标上高速推进
- AI 会抓错真源
- AI 会顺手扩范围（scope）
- AI 会出现幻觉把“看起来做完了”就当成“已经真的做完了”
- AI 会压缩上下文导致目标不清晰然后忘记规范
- AI 会猜 他注重的是实现这个结果，他并不知道错与对，实现了就是达成了
- AI 不会天然替你做回写和知识沉淀
- AI 不太擅长去复用已有的框架或者组件

## 什么叫控制面

我自己现在对“控制面”这个词的理解，已经不太会把它只理解成“一个放规范的地方”了。

如果一个仓库只是回答“这里有哪些资料”，那它本质上更像知识库。  
但如果一个仓库开始稳定回答下面这些问题，它才更接近我想说的控制面：

- 这次到底做什么，不做什么
- 哪些文档是当前真源，冲突的时候听谁
- 哪些改动可以直接做，哪些必须升级讨论或者走正式变更
- 开发执行前应该读什么，忽略什么
- PR 提交时必须证明什么
- 合并或者发布之后，哪些状态必须回写

说白了，我现在理解的控制面，管的已经不只是“资料存放”了，它更像是在管整个项目推进秩序。

如果再压缩一点，我现在理解的控制面，本质上就是把下面五件事收拢起来：

- 边界
- 真源
- 流程
- 证据
- 回写

只要这五件事还是分散在人脑、聊天记录、临场判断里，项目就还没有真正进入控制面状态。

## 先把整体工作流放出来看一下

我自己现在越来越觉得，在继续往下讲“控制面到底是什么”之前，最好先把整体工作流摆出来。  
因为脑子里如果没有一个全局图，后面再去讲目录分层、讲 `control-plane`、讲 `runtime`，读起来还是容易散。

### 整体闭环图

```text
PM 发起聚焦讨论
  -> 讨论收敛成决策
  -> OpenSpec / Spec 冻结边界和方案
  -> 拆任务
  -> 下发 Issue 到 FE / BE
  -> 开发执行
  -> PR 提交证据
  -> 审核合并
  -> 回写 runtime / risk / 经验
  -> 下一轮讨论
```

这条线其实就是我现在理解的项目主推进链路。

文档仓库在这里承接的，也早就不只是“存文档”这一个动作了，它其实把讨论、冻结、下发、执行、验证、回写这些关键动作都串起来了。

### 再看控制面各层的关系

```text
整个文档仓库 = 控制面

├─ 宪法层：docs/control-plane/
│  管范围、边界、治理红线、真源层级
│
├─ 流程层：docs/workflow/
│  管任务怎么进入、怎么拆、怎么发、怎么审、怎么回写
│
├─ 验证层：docs/harness/
│  管什么算验证完成，什么证据才有效
│
├─ 运行层：docs/runtime/
│  管当前阶段、任务状态、风险台账
│
└─ 支撑件：templates / contracts / openspec
   管统一模板、契约真源、正式变更机制
```

这两张图先放在这里，其实就是想让后面讲“控制面”这个词的时候，读的人脑子里先有一张整体图，不然很容易一会儿看目录，一会儿看流程，最后还是碎的。

## 这里说的控制面，要从整个仓库去看，不能只盯着 `docs/control-plane/` 这个目录

这一点我觉得特别容易误解，所以必须单独拎出来说。

很多人一看到 control-plane 这个名字，就会很自然地把“控制面”这三个字直接等同到这个目录上。

但我现在更想表达的是：`docs/control-plane/` 只是整个控制面里的宪法层，负责长期有效的边界、治理规则、范围冻结这些东西，而且这个地方一般是和项目强绑定的，每个项目都应该定制化。控制面这个词如果说完整一点，说的是整个文档仓库。

也就是说：

- 控制面的重点不落在某一个目录上
- 控制面是整个文档仓库一起工作形成的系统
- `docs/control-plane/` 只是其中一层
- `docs/workflow/`、`docs/harness/`、`docs/runtime/`、`templates/`、`contracts/`、`openspec/` 这些都属于控制面的组成部分

如果只盯着 `control-plane` 目录看，很容易把控制面理解成“一个放规则的地方”。但项目真想稳下来，光靠规则肯定不够，后面还得有流程、有验证、有运行态、有模板、有契约、有变更机制一起配合。

## 整个文档仓库，作为控制面，大概是怎么分层的

### 1. 宪法层

对应的是 `docs/control-plane/`。

这一层主要回答的是：

- 项目做什么，不做什么
- 哪些边界不能随便改
- 治理红线是什么
- 真源层级怎么裁决

这层东西应该尽量稳定，不应该混入太多阶段性状态。
类似于存放项目开始定下的PRD文档，然后与该项目必要的东西

### 2. 流程层

对应的是 `docs/workflow/`。

这一层回答的是：

- 需求怎么进入开发流
- 什么情况下要走 OpenSpec
- Spec 怎么拆成任务
- Issue 怎么下发
- PR 怎么审
- 发版怎么冻结
- 回写怎么做

换句话说，宪法层更像是在告诉你哪些边界动不得，流程层则是在把“那具体应该怎么走”讲清楚。

### 3. 验证层

对应的是 `docs/harness/`。

这一层回答的是：

- 什么才算真的验证过
- 什么证据才算有效
- 什么情况下可以合并
- 什么情况下可以发版

这一层很重要，因为如果没有它，前面的规则和流程最后还是会回到“凭感觉验收”。

### 4. 运行层

对应的是 `docs/runtime/`。

这一层回答的是：

- 当前阶段做到哪了
- 哪些任务已经下发
- 哪些任务卡住了
- 当前有哪些风险

这一层承接的更多是项目当前运行状态的面板，它和前面那些长期规则放在一起看，角色其实很不一样。

### 5. 支撑件

除了上面四层，我觉得还有三个支撑件不能忽略：

- `templates/`：统一 Spec、Issue、PR、Harness checklist 的写法
- `contracts/`：冻结跨端契约真源
- `openspec/`：让重大变更先冻结、再执行、最后归档

所以我更在意的，也不是某一个目录写得多完整，而是“分层规则 + 工作流 + 验证机制 + 运行面板 + 模板 + 契约 + 变更机制”这一整套东西有没有一起工作起来。

## 为什么这种分层是必要的

因为如果不分层，项目很容易陷入两种坏状态。

第一种坏状态是：固定规则和运行状态混在一起。

结果会是：

- 文档越来越脏
- 阶段性叙事被误读成长期规则
- AI 把短期状态当成稳定约束

第二种坏状态是：只有规则，没有运行态。

结果会是：

- 文档写得很完整
- 但没人知道现在推进到哪
- 项目真实进度还是黑箱
- 仓库越来越像档案馆，不像控制台

所以固定层和运行层分离，更多是在保证两件事能同时成立：长期规则保持稳定，短期状态又能灵活更新。

## 真源层级为什么这么关键

这一点在人类协作里可能还不一定是最先被注意到的，但在 AI 协作里，我觉得它已经接近基础设施了。

因为人看到多份文档冲突时，通常会追问：

- 哪个更新
- 哪个算数
- 这个是不是旧版本

但 AI 往往不会主动这么追问。它很容易把最近读到的、字面最完整的、语气最强的那一份东西当成真相。

所以，真源层级在我看来已经不只是文档整理习惯了，它其实是在明确告诉 AI：

当多份材料不一致的时候，到底应该听谁的。

这一点如果不明确，AI 很容易抓错依据，然后在错误依据上高效执行。

## 真正能统筹前后端推进的，其实是这条工作流

讲到这里，我觉得就可以进入最核心的一部分了。也就是：这个文档仓库到底是怎么统筹推进前后端项目开发的？

如果用我们文档仓库里真实的工作流去对齐，我觉得可以把它理解成下面这条闭环：

1. PM 发起聚焦讨论
2. 讨论收敛成决策
3. 通过 OpenSpec 或 Spec 冻结边界和方案
4. 拆出任务，明确依赖、优先级、归属仓库
5. 发布到任务池，进入 Ready
6. 对应仓库执行开发
7. 提交 PR，附上证据
8. 审核通过后合并
9. 回写运行态、风险、经验
10. 进入下一轮讨论

这个顺序其实和我们 `workflow-example.md` 还有 `task-release-and-sync-sop.md` 里写的东西是能对上的。

## 把这个工作流展开来看，每一步到底在防什么偏

### 第 1 步：先讨论，核心还是为了把问题收拢

PM 先发起讨论，倒不只是为了走一个流程动作，更多是因为很多需求在进入开发前其实还没有真正把边界收住。

比如：

- 这次到底做不做
- 是现在做还是后面做
- 做到一阶还是二阶
- 哪些属于本次范围，哪些明确不做

如果这一步不做，后面开发入口从一开始就是漂的。

### 第 2 步：把讨论收成决策，不要只停留在聊天里

很多团队真正的问题，经常不是没讨论，而是讨论完了以后，结论没有真正沉淀成后续执行的真源。

如果讨论只停留在聊天、会议、口头上，那下一步执行时，大家还是会各自理解。

所以控制面很重要的一点，就是要把“讨论过”变成“冻结了”。

### 第 3 步：用 OpenSpec 或 Spec 冻结边界

当一个需求要真正进入开发流的时候，就不能只靠一句“差不多就是这样”。

它至少要被收成：

- 目标
- 范围
- 非目标
- 验收标准
- 契约影响
- 风险和回滚口径

如果是边界内的小改动，可以按规则直接走。  
如果涉及范围、契约、门禁、治理边界，那就应该升级到 OpenSpec 这种正式变更路径。

这一层做的事情，本质上就是防目标漂移。

### 第 4 步：拆任务，不要让执行入口还是一句模糊的大目标

需求冻结之后，还不能直接丢给开发或者 AI 去干。

中间还要有一步：拆任务。

任务拆解至少要明确：

- 对应哪一部分 Spec
- 属于哪个仓库
- 有哪些前置依赖
- 验收标准是什么
- 当前处于流程的哪个位置

这其实就是把“大目标”翻译成“可执行单元”。

### 第 5 步：发布到任务池，进入 Ready

这一步我以前也容易忽略，后来越来越觉得重要。

因为任务写出来还不算真正进开发流，它还要满足进入 Ready 的最低门槛。

根据我们的 `task-release-and-sync-sop.md`，至少要有这些东西：

- 业务目标
- 范围边界
- 验收标准
- 依赖关系
- 优先级
- 所属流程位置

再加上对应的 `level / freeze / status` 这些标签，任务才算真正进入开发流，不会继续停留在“有人想做”这个层面。

### 第 6 步：Issue 自包含，下发真正可执行的上下文

这一点我觉得是 AI 时代特别重要的一步。

Issue 不能只是一个标题加一句描述，它应该尽量自包含。

也就是说，Agent 真正接到任务的时候，入口里就应该明确写清楚：

- 做什么
- 为什么现在做
- 真源是什么
- 非目标是什么
- 允许改哪里，不允许改哪里
- 需要怎么验证

这样做的好处很直接：

- 降低对聊天和口头解释的依赖
- 降低换人、换 Agent、跨轮接手的信息损耗
- 降低 AI 自己漫游仓库、抓错真源的概率

这一层，本质上是在防执行入口漂移。

### 第 7 步：开发执行，但这时候已经不能再靠自由发挥

到这里前后端仓库才真正开始写代码。

但这时候的开发，基本就不能再停留在“我理解一下需求然后自由发挥”这个状态了，更稳妥的方式会是：

- 按 Issue 给定的真源执行
- 按仓库 AGENTS.md 指定的读取顺序执行
- 按契约和范围边界执行
- 按 Harness 要求准备验证证据

也就是说，控制面做的事情并不是替开发写代码，它更像是在开发动手之前，就先把“怎么做才不容易偏”尽量前置出来。

### 第 8 步：PR 到这里更像是在提交证据

这一步我现在也越来越认可它的重要性。

PR 里真正有价值的，也不只是代码 diff，本质上还是看你能不能把下面这些东西交代清楚：

- 为什么做
- 为什么是现在
- 明确不做什么
- 关联的 Issue / Spec / Contract 是什么
- 跑了什么验证
- 结果是什么
- 证据是什么
- 风险和回滚是什么

如果没有这些，很多时候所谓评审，最后还是在靠感觉。

而 AI 场景下，“看起来做完了”和“真的验证过了”差别特别大。

### 第 9 步：审核和合并，不只是看功能像不像

传统项目里，很多时候合并更像是“看起来差不多能用”。

但控制面想做的事情，是把审核标准也外置出来。

也就是说 Reviewer 或 PM 审的，更多是下面这些东西有没有交代到位：

- 决策上下文清不清楚
- 真源关联齐不齐
- Harness 证据够不够
- Spec 场景覆盖全不全
- 有没有超范围修改
- 风险和回滚有没有交代

这一层，本质上是在防验收漂移。

### 第 10 步：回写，这一步我更愿意把它看成闭环真正成立的地方

很多时候大家最容易忽略的就是回写。

但我越来越觉得，回写很难把它当成附加动作看待，它更像是控制面里最有复利的一层。

因为如果合并之后不去更新：

- `docs/runtime/issue-dispatch.md`
- `docs/runtime/risk-register.md`
- `docs/runtime/phase-current.md`
- 必要时的新规则、新模板、新经验

那么这次开发虽然做完了，但组织并没有真正变得更强。

没有回写，流程只是一次性交付。  
有回写，项目才会越来越不像“每次重新来一遍”，而更像一个持续积累的系统。

## 为什么这套工作流比传统的“PM -> 开发 -> 验收”更强

传统流程更像：

PM 提需求 -> 开发自己理解 -> 开发提交结果 -> 大家一起看像不像

如果换一种更贴切的说法，这套控制面工作流更像：

PM 发起讨论 -> 冻结边界 -> 生成可执行任务 -> 指定真源 -> 要求证据 -> 回写系统记忆

差异不在于有没有 PM、有没有开发、有没有验收，而在于治理重心变了。

传统流程主要在管理：

- 谁在做
- 做没做完
- 结果像不像

控制面流程主要在管理：

- 为什么现在做这件事
- 这次做到哪里算完成
- 这件事现在推进到哪了
- 哪些不能顺手一起做
- 这次是依据什么做的
- 谁能复核它真的做完了
- 做完之后怎么进入下一轮组织记忆

对 AI 来说，后者比前者重要得多。

因为 AI 最擅长的是“在给定目标下高速展开”，最不擅长的是“自己判断组织语境里哪些隐含约束最重要”。

## 我现在最认可这套体系的几个价值

### 1. 它把模糊协作外置成了工程秩序

以前很多东西都在脑子里，现在它们被写成了边界、真源、流程、门禁、回写。

### 2. 它让项目不再只靠聊天推进

讨论当然可以有，但真正进入执行的那一层，还是应该落到已经冻结过的资产上，不然最后还是会散在聊天窗口里的理解里。

### 3. 它让前后端协作更容易围绕同一套依据

这样推进的时候，至少不会变成 FE 一套理解、BE 一套理解、PM 再一套理解，大家能更稳定地围绕同一份 Spec、Contract、Issue 和验收标准往前走。

### 4. 它让“验证”和“回写”真正进入主流程

也就不会等到做完之后才临时想要不要补，而是在一开始就要求你考虑：

- 怎么证明做完
- 做完之后怎么回到控制面

## 我也觉得这套体系有几个特别要警惕的点

### 1. 控制面自己也会漂

如果 README、AGENTS、契约路径、真源定义彼此不一致，控制面本身就会向 AI 发出冲突信号。

### 2. runtime 很容易成为最弱的一层

规则大家愿意写，但运行态大家不一定愿意长期维护。最后容易出现“规则很完整，实际推进还是黑箱”。

### 3. 回写最容易被跳过

因为它不直接产出功能，所以最容易在忙的时候被牺牲，但它偏偏又是最关键的闭环动作。

### 4. 证据可能流于形式

如果 PR 证据只是勾选框、口头说明、无法复核的截图，那流程看起来完整，实际上还是凭感觉合并。

### 5. 模板仓库可能被误解成“复制完就可以了”

`template-governance` 提供的更多是控制面骨架，离一个已经填满项目真相的控制面还有距离。如果只是复制结构，不补具体范围、系统设计、治理规则、契约和各仓库 AGENTS，那最后还是会是一个空壳。

## 最后收一下

如果让我现在用一句更直接的话来总结：

文档仓库之所以最后能长成控制面，在我看来也不是因为它把“写文档”这件事做得更重了，更多还是因为它把项目推进里最容易丢失、最容易口头化、最容易让 AI 误解的部分，提前固化成了可以被读取、被执行、被验证、被回写的工程秩序。

所以真正要守住的，也不只是文档数量，更重要的是这条秩序：

先讨论收束，再冻结边界；先明确真源，再开始执行；先给出证据，再完成合并；先完成回写，再进入下一轮。

我自己现在越来越倾向于相信，AI Native 项目里，真正重要的焦点也不只是“AI 能不能写”，更关键的是“我们有没有给它一套不容易越写越偏的控制面”。
